# Apex Spices SEO Audit and Implementation Plan

Audit date: 2026-07-19
Website: https://www.apexspices.lk/
Stack: React 19, Vite, React Router, Express, MongoDB, Vercel frontend and backend

## Executive summary

The storefront has a workable SEO foundation, but several production issues currently prevent it from getting the full value of its product and category content.

The five most important findings are:

1. The public sitemap at `https://www.apexspices.lk/sitemap.xml` is invalid for Google because every `<loc>` is relative. It contains only five URLs and omits all live products and categories. The complete dynamic sitemap exists on the API domain but is not the sitemap served by the storefront.
2. Most public routes ship the same generic homepage HTML and set metadata only after JavaScript runs. Only the shop, product, category, and three policy templates currently call the SEO utility. Social crawlers and search engines with delayed rendering can receive stale or generic metadata.
3. Product and category SEO still uses the former "Apex Link Group" positioning. Product data also contains serious mismatches: cinnamon uses a cardamom image and cardamom SKU, its origin says India, cardamom uses a cinnamon image, and the Product schema reports `USD 3500.99` while the page shows `LKR 3,500.99`.
4. Index controls are incomplete. Auth, cart, order, vendor, and utility routes do not send `noindex`; duplicate policy aliases self-canonicalize; and unknown URLs return HTTP 200 through the SPA rewrite, producing soft 404s.
5. Mobile performance needs work. Representative Lighthouse lab runs scored 80 for home, 63 for shop, and 70 for a product page. The shop CLS was 0.361 and product CLS was 1.0. The homepage LCP was 4.7 seconds. Google recommends LCP <= 2.5 seconds, INP <= 200 ms, and CLS <= 0.1.

Current opportunity: the domain is small enough that these issues can be corrected without a risky platform rewrite. The best near-term architecture is to keep the existing application, add server-rendered or prerendered HTML for public SEO routes, proxy the existing dynamic sitemap through the primary domain, and keep private/admin routes as a client-side SPA.

## Critical technical findings

### 1. The production sitemap is invalid and incomplete

Production currently serves this from the storefront:

```xml
<url><loc>/</loc></url>
<url><loc>/products</loc></url>
```

Sitemap `<loc>` values must be absolute. Lighthouse reports `Sitemap: /sitemap.xml` in `robots.txt` as an invalid sitemap URL as well. The storefront sitemap includes only `/`, `/products`, `/categories`, `/rfq`, and `/contact`.

The backend already generates the correct full set at `https://api.apexspices.lk/sitemap.xml`, including product/category `lastmod` values. The frontend static file shadows that useful endpoint.

Recommended implementation:

1. Remove the static `frontend/public/sitemap.xml` and proxy `/sitemap.xml` to the backend generator under the primary host.
2. Serve an absolute sitemap line in robots: `Sitemap: https://www.apexspices.lk/sitemap.xml`.
3. Remove `/rfq` while it renders an authenticated login experience.
4. Include only canonical, indexable, HTTP 200 URLs.
5. Add an image sitemap extension for product images when the catalog grows.

Example Vercel rewrite, placed before the SPA fallback:

```json
{
  "rewrites": [
    {
      "source": "/sitemap.xml",
      "destination": "https://api.apexspices.lk/sitemap.xml"
    },
    {
      "source": "/robots.txt",
      "destination": "https://api.apexspices.lk/robots.txt"
    },
    {
      "source": "/((?!assets/|sw\\.js$|.*\\.[^/]+$).*)",
      "destination": "/index.html"
    }
  ]
}
```

Verify that external rewrites are enabled in the Vercel project. A build-time sitemap generator is the fallback if they are not.

### 2. Public pages need server-rendered or prerendered SEO HTML

The initial HTML for every React route contains the homepage title and description. Page metadata is later injected by `applySeo()`, and most pages never call it. Google can render JavaScript, but Google explicitly says server-side rendering or prerendering remains a good idea for speed and crawler compatibility.

Recommended architecture:

- Keep the existing React SPA for cart, account, checkout, vendor, and admin experiences.
- Prerender static public routes at build time: home, products, categories, about, contact, FAQ, shipping, returns, privacy, and terms.
- Generate product and category HTML from the API during deployment, or render those routes through a lightweight Vercel server function with cached responses.
- Regenerate after product/category changes using a Vercel deploy hook or an on-demand revalidation endpoint.

Do not rely on `react-helmet` alone. It improves client navigation but does not solve the generic initial HTML delivered to non-JavaScript social crawlers.

### 3. Product SEO data is factually inconsistent

Current live examples:

| Product | Problem |
|---|---|
| Premium Ceylon Cinnamon Sticks | Slug is `organic-cinnamon-pods`; SKU is `SP-CRD-01`; main image is cardamom; origin is India |
| Premium Ceylon Cardamom Pods | Main image filename is `premium-ceylon-cinnamon-sticks.webp`; SKU is `SP-CIN-02` |
| Pure Cloves | Main image filename is `pure-turmeric-powder.webp` |
| Whole Black Peppercorns | Copy says Tellicherry and origin is Vietnam, while the broader site targets Sri Lankan spices |

The product SEO endpoint reports the cinnamon offer as `USD 3500.99`, while the visible page displays `LKR 3,500.99`. This can invalidate merchant rich results and create Merchant Center price mismatch warnings.

Required action before publishing keyword-focused copy:

- Audit every product name, slug, SKU, image, image alt, botanical identity, source country, weight, ingredients, price, and currency.
- Set a stored base currency per product or a single verified catalog currency. Do not infer it from an unrelated backend default.
- Only use "Sri Lankan," "Ceylon," "organic," "certified," or "export quality" where product records and documentation support the claim.
- Convert every schema image to an absolute HTTPS URL and deduplicate the main image.

### 4. Index controls and HTTP status behavior are incomplete

The SPA fallback serves HTTP 200 for unknown paths. React displays a 404 page only after rendering, so crawlers initially see a successful document. Invalid product/category IDs behave similarly.

Fixes:

- Return HTTP 404 for unknown public routes and missing/deactivated products/categories.
- Return 401/403 for authenticated resources as appropriate.
- Add `X-Robots-Tag: noindex, nofollow` to all private/transactional route responses.
- Remove private URLs from sitemaps.
- Do not disallow a URL in robots.txt if Google needs to crawl its `noindex`; robots blocking can prevent Google from seeing the noindex directive.

Routes that should be `noindex, nofollow`:

```text
/cart
/checkout
/login
/register
/forgot-password
/reset-password/*
/profile
/account
/privacy-center
/customer-experience
/vendor/onboarding
/vendor/dashboard
/thank-you
/order/*
/orders/*
/admin/*
```

`/track-order` may be `noindex, follow`; it is a branded utility, not a search landing page. `/rfq` should either become a public wholesale landing page or be removed from the sitemap and marked noindex while login-gated.

### 5. Duplicate aliases self-canonicalize

The same policy content is available at both short and `/pages/...` URLs:

- `/returns` and `/pages/refund-policy`
- `/privacy` and `/pages/privacy-policy`
- `/terms` and `/pages/terms-and-conditions`

Because the SEO utility defaults to `window.location.href`, each duplicate claims itself as canonical. Add permanent 308 redirects from `/pages/...` to the short URLs. Keep only the short URLs in internal links and sitemaps.

### 6. Stale brand/topic language weakens relevance

The site still contains "Apex Link Group," "multi-industry marketplace," textiles, technology, industrial equipment, and cross-industry testimonials in customer-facing and SEO code. This conflicts with the Apex Spices brand and dilutes topical focus.

The stale language appears in `frontend/src/utils/seo.js`, `backend/controllers/seoController.js`, product/category fallbacks, product trust copy, FAQ, shipping, login/register, and homepage testimonials.

Replace it with accurate Apex Spices language and remove invented or unsupported testimonials. Real customer testimonials should identify the source and permission status; otherwise omit them.

## Heading hierarchy

General rule: one descriptive H1 per indexable page, H2 for major page sections, and H3 for cards/items within those sections. Do not choose headings based on font size.

Required changes:

- Home: change the generic H1 "Elevate Your Culinary Experience" to a descriptive spice-focused H1. Use the current phrase as supporting copy.
- Shop: add a visible H1; the live page currently has zero H1 elements.
- Category: change `Discover products in {category}` from H2 to H1; the live category page has zero H1 elements.
- Product: existing product-name H1 is correct. Product Details, Reviews, Trust, Related Products, and Recommendations should be H2. Trust-card names should be H3.
- Categories/About/Contact/FAQ/Policies: current one-H1 pattern is structurally sound, but revise the H1 wording according to the table above.
- Footer navigation labels do not need heading tags. Use styled `<p>` elements or keep a consistent footer-only hierarchy so it does not interfere with the page outline.

### Product schema

Correct the current Product builder so it:

- Uses the same price and currency shown to an anonymous Google visitor.
- Emits absolute, deduplicated image URLs.
- Includes `seller`, `url`, `sku`, valid GTIN or MPN where available, `weight`, and `category`.
- Includes shipping and return information either at Offer or OnlineStore level.
- Includes `aggregateRating` only when approved reviews exist.
- Includes individual Review nodes only for visible, approved reviews. Never mark unverified reviews as verified.
- Uses `ProductGroup` and `hasVariant` if pack sizes have separate prices/SKUs.

```js
const absoluteUrl = (value) => new URL(value, SITE_URL).href;
const images = [...new Set([product.image, ...(product.images || [])].filter(Boolean))]
  .map(absoluteUrl);

const productSchema = {
  '@context': 'https://schema.org',
  '@type': 'Product',
  '@id': `${canonicalUrl}#product`,
  name: product.name,
  description: product.description,
  image: images,
  url: canonicalUrl,
  sku: product.sku,
  brand: { '@type': 'Brand', name: 'Apex Spices' },
  category: product.category,
  offers: {
    '@type': 'Offer',
    url: canonicalUrl,
    price: Number(product.price).toFixed(2),
    priceCurrency: product.currency || 'LKR',
    availability: product.countInStock > 0
      ? 'https://schema.org/InStock'
      : 'https://schema.org/OutOfStock',
    itemCondition: 'https://schema.org/NewCondition',
    seller: { '@id': 'https://www.apexspices.lk/#store' }
  }
};
```

### Breadcrumb and collection schema

Add visible breadcrumbs and matching `BreadcrumbList` to category and product pages:

```text
Home > Categories > Whole Spices > Premium Ceylon Cinnamon Sticks
```

Add `ItemList` to the shop and category pages using the canonical product URLs in displayed order. Pagination must use crawlable links with unique `?page=2` URLs. Current Previous/Next controls are buttons that only mutate state; Google generally does not click buttons to discover later products.

### FAQ schema

The FAQ page can use `FAQPage` because questions and answers are visible. Google currently limits FAQ rich-result display mainly to well-known government and health sites, so the schema is useful for machine understanding but should not be expected to produce a commercial FAQ rich result.

### Review schema

The review database implementation correctly limits aggregate counts to approved verified-purchase records. Preserve that source of truth. Ensure the Product API, SEO API, cards, detail page, Merchant feed, and JSON-LD all read the same computed values. When there are no approved reviews, omit `aggregateRating` and `review`; display `0 reviews` in the UI.

## Canonical and URL strategy

### Immediate canonical rules

- Canonical host: `https://www.apexspices.lk` (the apex domain already redirects to `www` with 308).
- Add a self-canonical to every indexable page in the initial/prerendered HTML.
- Strip tracking parameters (`utm_*`, `gclid`, `fbclid`) from canonicals.
- Shop filter/sort URLs should canonicalize to the unfiltered collection unless a filter is intentionally built as a useful landing page.
- Paginated pages should have self-canonicals, not all canonicalize to page 1.
- Redirect duplicate policy aliases.

### Product URL improvement

The current `/product/:id` URL is stable but not descriptive. The model already stores a slug. After cleaning duplicate/incorrect slugs, migrate to:

```text
/products/premium-ceylon-cinnamon-sticks
/products/premium-ceylon-cardamom-pods
/products/pure-cloves
/products/whole-black-peppercorns
```

Implementation order:

1. Add a unique indexed `slug` field and a backend product-by-slug endpoint.
2. Render and canonicalize the slug route.
3. Permanently redirect every old ID URL to its slug URL.
4. Update all cards, breadcrumbs, related links, sitemap entries, Merchant feed links, and internal analytics.
5. Never reuse an old product slug for a different product.

Do not change URLs until the redirects and slug data are ready; correct canonicals and content on the current ID URLs first.

## Image SEO and media performance

Findings:

- Most images have alt attributes, which is good.
- The homepage LCP is a CSS background. Google Images does not index CSS background images, and Lighthouse cannot discover it early from the HTML.
- Product/category images do not consistently declare intrinsic `width` and `height`, contributing to layout shift.
- The 1024x1024 header logo is about 221 KB while displayed near 48x48.
- Lighthouse estimates about 755 KB image savings on home, 215 KB on shop, and 469 KB on the product page.
- Pexels category images are served at 1400 px for much smaller rendered sizes.
- Product filenames and visual subjects are mismatched for three of four products.

Implementation:

- Replace hero CSS backgrounds with a semantic `<picture><img>` using AVIF/WebP sources, `fetchpriority="high"`, `loading="eager"`, explicit dimensions, and responsive `srcset/sizes`.
- Keep only the true LCP image eager. Lazy-load below-fold category/product images.
- Generate widths around 320, 480, 768, 1024, and 1440 where appropriate. Cloudinary can supply format, quality, crop, and width transformations.
- Generate a 96x96 or 128x128 header logo asset; reserve 512x512 for app/organization use.
- Add `width` and `height` or `aspect-ratio` to every product/category/gallery image.
- Use descriptive filenames such as `ceylon-cinnamon-sticks-1000g.webp` and alt text such as `Ceylon cinnamon sticks and bark pieces, 1000g pack`.
- Product gallery alt text should identify the view, not repeat `gallery` for every image.
- Use original product photography wherever possible. Competitor pages win trust with origin, grading, packaging, and specification imagery.

## Core Web Vitals and performance

Mobile Lighthouse lab results captured during this audit:

| Template | Performance | SEO | FCP | LCP | TBT | CLS | Transfer |
|---|---:|---:|---:|---:|---:|---:|---:|
| Home | 80 | 92 | 1.7 s | 4.7 s | 140 ms | 0.001 | 1,663 KB |
| Shop | 63 | 92 | 1.7 s | 3.8 s | 320 ms | 0.361 | 423 KB |
| Product | 70 | 92 | 1.7 s | 1.9 s | 200 ms | 1.000 | 677 KB |

These are repeatable lab diagnostics, not CrUX field data. Search Console field data should be the ranking-oriented source after sufficient traffic exists.

Priority performance work:

1. Prerender route content so loading shells do not expand into full pages after API responses.
2. Give loading placeholders the same dimensions as final content. Product and shop layout currently push the entire footer/content after data arrives.
3. Make the home LCP image discoverable in initial HTML and add `fetchpriority=high`.
4. Deliver responsive category/product images and right-size the header logo.
5. Keep Stripe and checkout code isolated to checkout chunks; current route lazy loading is already a good foundation.
6. Cache public image files. `frontend/vercel.json` currently sends `no-store` for all non-`/assets` files, including public WebP/JPG/PNG images. Use versioned filenames with long cache headers or a moderate cache plus `stale-while-revalidate`.
7. Measure `web-vitals` in production and send LCP/INP/CLS to GA4 or the existing analytics endpoint with route/template dimensions.

Suggested public media header:

```json
{
  "source": "/((?:.*)\\.(?:webp|avif|png|jpe?g|svg|woff2))",
  "headers": [
    {
      "key": "Cache-Control",
      "value": "public, max-age=604800, stale-while-revalidate=86400"
    }
  ]
}
```

Use `immutable, max-age=31536000` only when the filename changes whenever the content changes.

## Analytics, Search Console, and Merchant Center

### Current state

- `VITE_GA_MEASUREMENT_ID` support exists in `frontend/src/utils/analytics.js`.
- The live rendered homepage did not contain a Google tag, Meta Pixel, TikTok tag, LinkedIn tag, or Google site verification meta tag.
- The consent UI stores preferences, but analytics/marketing scripts are initialized before consent logic and do not read those preferences.
- GA custom events exist for product view, cart, checkout, and purchase, but item payloads do not fully follow GA4 recommended ecommerce format.

### Required setup

1. Create/confirm a Search Console Domain property for `apexspices.lk` using DNS verification. Add the `https://www.apexspices.lk/` URL-prefix property for focused reporting if useful.
2. Submit only `https://www.apexspices.lk/sitemap.xml` after fixing it.
3. Inspect home, shop, one category, and each product template. Test live URL and rendered HTML.
4. Monitor Page Indexing, Core Web Vitals, Product snippets, Merchant listings, HTTPS, and Manual Actions reports.
5. Configure `VITE_GA_MEASUREMENT_ID` in the production frontend and verify requests in GA4 DebugView/Realtime.
6. Send explicit SPA `page_view` events on React Router location changes.
7. Use GA4 ecommerce events and the `items` array: `view_item_list`, `select_item`, `view_item`, `add_to_cart`, `view_cart`, `begin_checkout`, `add_shipping_info`, `add_payment_info`, `purchase`, and `refund`.
8. Use `transaction_id`, value, tax, shipping, currency, coupon, and item identifiers on purchase; deduplicate purchases by transaction ID.
9. Wire analytics and marketing script loading to stored consent, with Consent Mode where legally applicable.
10. Link GA4 to Search Console and Merchant Center.
11. Create Merchant Center, verify/claim the domain, enable free listings, and submit a product feed generated from the same source as the storefront.

Do not launch Merchant Center until price, currency, availability, image, shipping, return, and product identity data match exactly between the feed, structured data, and visible landing page.

## International SEO

- Keep the current English site globally accessible; do not use IP-based redirects.
- No `hreflang` is needed while there is only one English URL per page.
- If country versions are later created, use stable routes such as `/en-lk/`, `/en-gb/`, or `/en-us/`, self-canonicals, reciprocal hreflang, and `x-default`.
- Currency selection alone is not a reason to create indexable duplicate URLs. Google and Merchant Center must see a stable currency/price combination on each landing URL.
- Add country coverage, duties/taxes, shipping lead times, Incoterms, MOQ, packaging, certificates, and documentation to a real wholesale page.
- Build Merchant Center feeds per target country/currency rather than silently changing schema values from local storage.

## Verification checklist

Run this checklist after each SEO release:

- `curl`/View Source shows unique title, description, canonical, robots, OG, Twitter, and JSON-LD before JavaScript.
- Every sitemap URL is absolute, canonical, indexable, and returns HTTP 200.
- Unknown/deactivated URLs return 404 or 410.
- Private routes return `X-Robots-Tag: noindex` and are not in the sitemap.
- Rich Results Test passes Product and Breadcrumb markup.
- Schema.org validator passes non-Google schema types.
- Merchant Center feed price/currency/availability/image match the page exactly.
- Search Console URL Inspection rendered HTML matches the intended page.
- Lighthouse mobile runs for home, shop, category, and product show CLS < 0.1 and improving LCP.
- No broken internal links or empty categories are present.
- GA4 DebugView records page_view, view_item, add_to_cart, begin_checkout, and one deduplicated purchase.
